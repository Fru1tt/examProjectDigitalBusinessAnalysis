# 06 Project Plan (Retroactive, Final Deliverable Version)
 Our project is called *Predicting Customer Channel Preference for Better Resource Allocation*, and the team members are Bertine Josefsen and Carl Henrik Broch Grude.

Our starting point is a practical business problem. Retailers need to decide how much to invest in online channels, physical stores, and omni-channel solutions, but they often do not have a reliable way to understand customer channel preference. If they make wrong assumptions, they risk spending marketing budgets in the wrong places and missing opportunities to improve customer experience. The purpose of our project is therefore to build a decision-support solution that predicts likely channel preference for individual customers and turns this into useful recommendations.

The data source we plan to use is the *Online vs In-Store Shopping Behaviour Dataset* from Kaggle. The file in our project contains around 11,789 observations and 25 columns. It includes demographic variables, digital behavior variables, purchase behavior variables, and a target variable called `shopping_preference` with three classes: `online`, `store`, and `hybrid`. This dataset is suitable for our question because it includes both customer profile information and an explicit preference label that can be modeled.

We choose a model-based approach because manual rule-based logic is not enough for this kind of problem. A simple rule like “high internet hours means online preference” may sound reasonable, but real customer behavior depends on many variables at the same time. Some customers are digitally active but still buy mostly in-store. A machine learning model can learn combined patterns across many variables and estimate probabilities for each channel option. This gives a stronger and more realistic basis for business decisions. We still include a simple baseline model in the plan so we can verify that the advanced model actually adds value.

Our planned work process begins with technical setup and project structure. We will organize the repository with separate folders for raw data, processed data, scripts, outputs, and documentation. We will also set up version control and a changelog so every major change is traceable. This is important both for collaboration and for exam transparency.

After setup, we plan to prepare the data in a reproducible way. Instead of cleaning manually, we will build a preprocessing script that loads raw data, checks structure and value ranges, standardizes categories, and creates a clean analysis dataset. We will also generate a small quality report so it is clear what checks were done. This step is important because model quality depends on data quality.

The next planned phase is model training and comparison. We will train multiple models, including a baseline model and stronger classification models, and evaluate them on unseen test data. We will use accuracy, macro F1, weighted F1, class-level precision/recall/F1, and confusion matrices. We choose these metrics because the class distribution is imbalanced, with many more `store` cases than `hybrid` cases, so macro-level and class-level metrics are needed to avoid misleading conclusions.

Because the `hybrid` class is underrepresented, we plan a dedicated improvement step for minority-class performance. In this phase we tune the model decision threshold for hybrid probability and compare results before and after tuning. This gives a more fair and useful model for omni-channel analysis and improves the practical relevance of recommendations.

When the technical model is stable, we plan to convert outputs into business-facing material. This includes KPI tables, model comparison summaries, channel mix views, driver analysis, and segment-level recommendation tables. The reason for this step is simple: decision-makers need clear action-oriented outputs, not only technical performance numbers.

We also plan to create an interactive dashboard as a presentation and decision-support interface. In the dashboard, users can input a customer profile and receive predicted channel preference with probabilities. The dashboard will include an adaptive recommendation layer that separates channel preference from digital susceptibility. This is important because a customer can prefer store purchases but still be very reachable through digital marketing. We also plan a scenario comparison function so users can see how recommendation strategy changes when key behavior variables are adjusted.

The final phase of the plan is report writing. We will document the business problem, method, results, limitations, and recommendations using the outputs from the scripts and dashboard. We will keep references and assumptions explicit so the report is easy to follow and academically transparent.

Our planned timeline is seven weeks. In week one we complete setup and framing. In week two we complete data preparation. In week three we run baseline and initial model comparison. In week four we focus on hybrid-class improvement. In week five we produce executive outputs. In week six we finalize the dashboard and scenario comparison. In week seven we complete the final report and submission package.

Team responsibility is shared across the project, but with practical ownership by task area. One member takes primary ownership of data preparation and documentation drafting, while the other member takes primary ownership of modeling and dashboard implementation. Interpretation of results, recommendations, and final quality checks are done together.

We define project success as delivering a full end-to-end pipeline from raw data to recommendations, showing that the model performs better than a simple baseline, clearly explaining key preference drivers, and presenting results in a format that business users can understand. A successful outcome also requires a usable dashboard and a coherent final report.

There are important risks. The strongest risk is class imbalance, which can hide poor performance for smaller classes if we only look at accuracy. We handle this by using macro and class-level evaluation and by tuning the hybrid threshold. Another risk is that the data is simulated, not from one specific retailer, so we will clearly state this as a limitation and avoid over-claiming external validity. A third risk is that outputs can become too technical, which we address by preparing executive tables, visual summaries, and plain-language recommendations. A final risk is time pressure, which we handle by dividing work into script-based modules and clear weekly milestones.

The planned final deliverables are a cleaned dataset and quality report, reproducible scripts for preparation, modeling, and output generation, model evaluation with hybrid-class improvement, business-ready figures and tables, an interactive dashboard, and a complete written report with references and documented limitations.
